# Plan: внедрение практик тестирования (SmartGraphical)

Пошаговый план расширения автоматических тестов с опорой на слои: контракт адаптера, юниты правил на синтетической модели, узкие end-to-end проверки и легкая дисциплина покрытия задач (`task_id`).

## Цели

- Раньше находить регрессии на самом стабильном слое (`NormalizedAuditModel` + правила).
- Развести ответственность: корректность адаптера отдельно от логики правил и от HTTP/CLI.
- Не привязываться к хрупким полным текстам отчётов там, где достаточно инвариантов.
- Видеть пробелы: какие правила/задачи не покрыты тестами намеренно, а какие забытые.

## Текущая опора в проекте

- Юнит-тесты правил часто собирают синтетическую `NormalizedAuditModel` в Python и вызывают `run_*` (быстро, детерминированно).
- Интеграция покрывает адаптер Solidity и полный pipeline на эталонном контракте (`SimpleAuction.sol`), с проверками-инвариантами по выдаче `findings`.
- Отдельно идут пакеты правил для C/Node; HTTP и история имеют контрактные тесты.

## Фаза 1: матрица правил (unit, уровень модели)

**Задача:** у каждого автоматизируемого Solidity-правила — минимум один positive и один negative кейс на искусственном контексте.

**Шаги:**

1. Сверить список правил (`smartgraphical/core/rules/` и смежное) с `tests/unit/test_rules_*.py`.
2. Для пробелов добавить тесты:
   - фиксировать стабильные `rule_id` / `task_id`;
   - по возможности проверять подстроки в `message` и структуру `evidences`, не дословную простыню текста.
3. Вынести общие фабрики контекстов (общий модуль под `tests/` по согласованию с командой) вместо копипасты `_context_with_*`.
4. В модуле правила — короткая отсылка к тесту (`see tests/unit/...`).

**Критерий готовности:** таблица или чеклист `rule_id -> файл теста -> positive/negative`; CI без новых flaky-тестов.

## Фаза 2: фикстуры адаптера (фрагменты Solidity)

**Задача:** маленькие зафиксированные `.sol` файлы, по которым проверяются стабильные факты нормализации, без привязки к выдаче правил.

**Шаги:**

1. Каталог фикстур, например `tests/fixtures/solidity/` — один сценарий на файл (guards, external calls, state, граничные случаи читателя).
2. Для каждой фикстуры — тесты на имена типов/функций, подмножества `guard_facts`, `external_calls`, `state_entities`, ключи `evidence_index` — в рамках того, что адаптер сегодня гарантирует.
3. В docstring теста фиксировать известные ограничения («bool state сейчас не в state_entities»), чтобы осознанно обновлять ожидания при расширении reader.

**Критерий готовности:** 3–5 фикстур с явными ожиданиями; падение теста = осознанное изменение модели в том же PR.

## Фаза 3: end-to-end инварианты (мало, но жёстко)

**Задача:** несколько полных проходов adapter + engine (+ при необходимости API), без лока всего отчёта.

**Шаги:**

1. Сохранить/укрепить текущий стиль pipeline: допустимое множество `rule_id`, обязательные поля finding, evidence, политика дубликатов.
2. Добавить не больше 1–2 эталонных контрактов только если они дают другой путь парсера (несколько контрактов, библиотека и т.д.), недостижимый юнитами.
3. При необходимости — короткий API-тест ingest + analyze на форму JSON (список `findings`, обязательные ключи).

**Критерий готовности:** тяжёлых E2E не больше ~5 сценариев; без snapshot всего Markdown-отчёта.

## Фаза 4: согласование с каталогом задач

**Задача:** новая задача в UI/API не попадает «мимо» тестов молча.

**Шаги:**

1. Снять список `task_id` / групп из реестра или HTTP-слоя.
2. Для каждой задачи пометить: покрыта model-level тестом, только E2E-инвариантами, или освобождена (documentation-only / visualization-only) с причиной в доке.
3. По желанию позже — тест-дверь: новый id без строки в декларативном файле (`TASK_COVERAGE.yaml`) ломает CI.

**Критерий готовности:** один источник истины (таблица в docs или YAML), проверка в чеклисте PR для новых правил.

**Сделано для Solidity:** `tests/fixtures/solidity_task_coverage.json` и `tests/unit/test_solidity_task_coverage_declared.py` сверяют JSON с `build_rule_registry()`, слагами правил и `web_api.list_tasks("solidity")`.

**Сделано для C:** `tests/fixtures/c_task_coverage.json` и `tests/unit/test_c_task_coverage_declared.py` — то же для `build_c_rule_registry()` и `web_api.list_tasks("c")` (матрица: `docs/testing_c_rule_coverage_matrix.md`).

## Фаза 5: снимки и производительность (опционально)

**Задача:** точечно повысить уверенность без шума в diff.

**Шаги:**

1. Снимки только узких структур (JSON куска модели), не полного scan output.
2. Маркер `integration_large` / ночной прогон с грубым лимитом времени на самой большой фикстуре.

**Критерий готовности:** обычный CI остаётся быстрым; тяжёлое — по маркеру или расписанию.

**Сделано (минимально):**

- Общая функция `narrow_normalized_model_shape_json` в `tests/integration/pipeline_invariant_helpers.py`.
- Узкий снимок имени языка/basename типа/spисков функций и `state_entities` для `MinimalGuard.sol` и `WithdrawNoGuard.sol`: `tests/integration/test_solidity_adapter_fixtures.py` (`*_phase5_shape_snapshot`). Аналогично для C: `MinimalTu.c`, `FloatToUintCast.c` в `tests/integration/test_c_adapter_fixtures.py`. Формат: `json.dumps(..., sort_keys=True, separators=(",", ":"))` без пробелов.
- Прогон **integration_large**: класс `IntegrationLargePipelineBudgetTests` в `tests/integration/test_full_pipeline.py`. По умолчанию **пропуск** (`skip`). Включение: переменная окружения **`SMARTGRAPHICAL_RUN_INTEGRATION_LARGE=1`**; лимит секунд — **`SMARTGRAPHICAL_PIPELINE_BATCH_BUDGET_SEC`** (по умолчанию `60`) на партию из четырёх Solidity-фикстур подряд (`analyze` + `run_all` на каждой).

## Порядок внедрения

1. Фаза 1 — максимальный эффект при текущей архитектуре.
2. Фаза 2 — защита от «тихой» поломки адаптера.
3. Фаза 3 — страховка стыков.
4. Фаза 4 — процесс и видимость.
5. Фаза 5 — по боли.

## Дисциплина PR

- Новое правило: в том же PR — тесты фазы 1; если нужны новые факты модели — фикстура/расширение фазы 2.
- Изменение адаптера: прогон фазы 2 + затронутые тесты фазы 1; правки ожиданий с комментарием «почему».
- Релиз: зелёный минимальный E2E-набор; тяжёлое — по политике CI.

## Риски

- Дублирование model-тестов и E2E — держать E2E коротким.
- CRLF и пути — через `.gitattributes` и аккуратные пути в тестах.
- Граф: проверять `available`/форму данных, не визуал.

## Метрики успеха (3–6 месяцев)

- Меньше случаев «правило перестало срабатывать» без падающего теста.
- Новые контрибьюторы копируют фабрики/фикстуры, а не изобретают форматы.
- Время CI в приемлемых пределах (порог задаёт команда).

## Что дальше (бэклог после фаз 1–4)

Фазы **1–4** для **Solidity** и **C** закрыты в виде кода и манифестов; дальше по приоритету:

2. **Стабильность существующего набора:** привести к зелёному состоянию тесты, которые падают из‑за окружения (пути, отсутствие `SimpleAuction.sol`, опциональный FastAPI), если они входят в обязательный CI.
