# План: многофайловый анализ и единый граф

Пошаговый план внедрения анализа нескольких исходников за один запуск и выдачи одного объединённого графа (и при необходимости одного набора findings). Примеры строк в коде/контрактах — ASCII.

## 1. Цель и границы

**Цель:** пользователь выбирает несколько файлов (или архив), запускает один скан и получает **один граф** (и findings), отражающий связи между сущностями из разных файлов там, где это технически возможно.

**Границы (зафиксировать в ТЗ):**

- **Solidity:** без полноценного конвейера компиляции межфайловые связи по `import` и общим типам могут быть неполными. Описать гарантии (например: узлы/рёбра внутри каждого файла + явные вызовы, видимые адаптеру).
- **C:** TU против целого проекта; выбрать: либо несколько подграфов в одном представлении, либо реальное слияние после общего фронта (сложнее).

## 2. Модель данных и API

### 2.1. Сущность «пакет»

Ввести **project / bundle** (или расширить artifact):

- один **логический артефакт** с **несколькими** путями (`artifact_files`: порядок, относительное имя, `sha256`, роль: `primary` / `support`);
- один **scan** по bundle — один `graph.json` / один идентификатор прогона.

### 2.2. HTTP

- либо расширить `POST /artifacts`: `multipart` с несколькими полями `file` или одним `archive` + опциональный manifest;
- либо добавить `POST /projects` с тем же телом, чтобы не менять контракт «один файл».

### 2.3. Обратная совместимость

Сценарий «один файл» сохраняется: bundle из одного файла = текущее поведение.

## 3. Пайплайн анализа

### 3.1. Сохранение

Класть файлы в `workspace/artifacts/<id>/...`, сохранять относительные пути (из zip или имён), избегать коллизий.

### 3.2. Языковой слой

- **Вариант A (быстрый):** для каждого файла — существующий адаптер, локальные нормализованные модели, затем **слияние** в одну модель с префиксами id и метаданными пути (`file_id` / `path`).
- **Вариант B (качественнее):** один проектный разбор/индекс по всем путям — больше работы, лучше межфайловые рёбра.

Рекомендация: **фаза 1 — A**, фаза 2 — точечные улучшения для импортов, где адаптер это позволяет.

### 3.3. Слияние графа

- единое пространство имён узлов: стабильный ключ = hash(path + kind + name + сигнатура) или явная схема из документации графа;
- рёбра: без дублирования; конфликты — политика «последний выигрывает» или объединение атрибутов;
- в сериализаторе: поле `source_file` (или эквивалент) на узлах/рёбрах для UI.

### 3.4. Findings

Решить: плоский список с полем `file` или группировка по файлу в API и на UI.

## 4. Фронтенд

- загрузка: multi-select или zip; лимиты (число файлов, суммарный размер);
- страница скана: тот же граф; опционально фильтр «только файл X», визуальная группировка по файлу;
- история / diff: сравнение сканов **одного** bundle id.

## 5. Документация и контракт

- обновить `docs/graph_schema_logic.md`: семантика multi-file, `source_file`, правила id при merge;
- тесты HTTP-контракта: новые поля и ошибки (слишком много файлов, битый zip).

## 6. Этапы внедрения

| Этап | Содержание |
|------|------------|
| **M0** | ТЗ: один vs несколько файлов, лимиты, формат zip/manifest. |
| **M1** | Хранилище: схема БД + репозиторий для multi-path artifact; раскладка на диске. |
| **M2** | API: приём нескольких файлов, создание bundle; опционально промежуточный шаг — N графов в одном JSON-массиве. |

| **M5** | Усиление межфайловых рёбер (язык-специфично). |

## 7. Риски

- **Память и время:** крупные проекты — батчинг или лимиты;
- **Безопасность zip:** zip slip, лимит распаковки;
- **Детерминизм:** порядок файлов влияет на стабильные id — зафиксировать сортировку по пути.
